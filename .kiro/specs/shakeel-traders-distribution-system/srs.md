## 1. Introduction & Purpose

This document defines the complete software requirements for Shakeel Traders' Distribution Order System — a fully offline, locally hosted solution to manage daily distribution operations. The system replaces the existing Salesflo software and is tailored precisely to Shakeel Traders' workflows.

### 1.1 Three Sales Channels

The system manages three distinct and separately tracked sales channels:

- **Order Booker Sales** — orders booked in the field via mobile app, converted to bills by admin
- **Salesman (Van) Sales** — stock issued to salesman, sold directly from the van, returned stock settled in the evening
- **Direct Shop Sales** — bills created instantly by admin from the web panel

### 1.2 Key Design Principles

- 100% offline capable for mobile app operations after morning sync
- Single local office computer hosts the entire system — no cloud dependency
- All three sales channels tracked separately in every dashboard and report
- Order booking and cash recovery are independent tasks — admin decides who does what

## 2. System Overview

The entire system runs on one local office computer. There is no cloud, no internet dependency for daily operations.

| Component | Technology / Description |
|---|---|
| Web Admin Panel | Node.js + Express + EJS + Bootstrap, MVC Architecture |
| Database | MySQL hosted on the local office computer |
| Mobile App | Flutter (Android) |
| Mobile Local DB | sqflite or Hive for offline storage |
| Deployment | Single local LAN — no internet required |
| Backup | Automated daily MySQL dump, configurable time, admin-restorable |

Mobile app users must enter the server IP and port on first launch and test the connection before login. All daily operations work fully offline after morning sync.

## 3. User Roles

There are exactly four role types in this system, with distinct access boundaries:

| Role | System Access | Responsibilities |
|---|---|---|
| Admin | Web panel only | Full access: stock management, order approval, bill generation, salesman settlement, cash management, supplier management, reports, salary management |
| Order Booker | Mobile app only | Books orders in the field; records cash recoveries from assigned bills on same day they are assigned. Order booking and cash recovery are SEPARATE tasks. |
| Salesman | Mobile app only | Enters issued stock in the morning; enters returned stock in the evening. No shop assignment. |
| Delivery Man | No system login | Physical staff member. Admin manages salary, advances, clearance via web panel. Admin records cash collected by delivery man. |

## 4. Core Business Concepts & Terminology

| Term | Definition |
|---|---|
| **Route** | A named collection of shops grouped geographically. Routes replace what was previously called "areas" or "cities." A route is the unit of work assignment for order bookers. |
| **Route Assignment** | Admin assigns one or more routes to an order booker per day for ORDER BOOKING only. Cash recovery assignment is completely separate. |
| **Shop** | A retail or wholesale outlet belonging to exactly one route. Every shop has a running ledger. |
| **Product** | Every product has a unique mandatory SKU Code. Tracked in cartons and loose units. Has retail price and wholesale price. |
| **Supplier / Company** | Source of stock (e.g., CBL). Each supplier has a completely separate ledger for advances, stock receipts, and claims. |
| **Order** | Created by an order booker in the field. Synced in the evening. Must be approved and converted to bills by admin. |
| **Bill** | Generated by admin from an approved order, or created directly by admin for a Direct Shop Sale. |
| **Salesman Stock Issuance** | Morning record of stock issued to a salesman. Admin must verify before warehouse stock is deducted. |
| **Salesman Stock Return** | Evening record of unsold stock returned. Admin verifies. Net sold = issued minus returned at retail price. |
| **Cash Recovery** | Collection of outstanding bill payments. Admin assigns specific bills to an order booker independently of route assignment. The order booker visits those shops and records collected amounts. |
| **Shop Advance** | Payment made by a shop in advance. Auto-deducted from new bills until exhausted. |
| **Supplier Advance** | Money paid to a supplier before stock arrives. Stock receipt deducts from this balance. |
| **Claim** | Complaint or return to a supplier. Tracked for records but NOT added to warehouse stock. Cleared claims add value back to supplier advance. |
| **Centralized Cash Screen** | Shows all cash received: salesman sales, order booker recoveries, delivery man same-day bill collections. |
| **Delivery Man** | Physical staff — no system login. Takes printed bills to shops. Admin records his cash collections. |

## 5. Business Rules (Master List)

| Rule ID | Rule Description |
|---|---|
| **BR-01** | System operates on one local office computer. No internet required for daily operations. |
| **BR-02** | Mobile app must show a "Connect to Server" screen before login. Requires IP address and port. Test Connection button must succeed before login is accessible. |
| **BR-03** | Every product must have a unique SKU Code. SKU is the primary product identifier. |
| **BR-04** | Stock is organized as Routes. A route contains one or more shops. Route assignment is for ORDER BOOKING ONLY and is separate from cash recovery assignment. |
| **BR-05** | Warehouse stock can never go negative under any circumstance. |
| **BR-06** | Salesman stock issuance must be verified by admin before warehouse stock is deducted. |
| **BR-07** | Salesman stock return must be verified by admin before: (a) return stock is added back to warehouse, and (b) net sale value is finalized. |
| **BR-08** | Salesman sales value is calculated by the system at retail price per product sold. Admin can review and edit before final approval. |
| **BR-09** | Salesman has no assigned shops. Sales value is calculated from (issued minus returned) quantities × retail price. |
| **BR-10** | On evening sync, if a product is completely out of stock it is removed from the order. If partial stock is available, the quantity is reduced to available amount. Order booker receives an itemized notification of all changes. |
| **BR-11** | Orders are converted to bills one-to-one (one bill per order). Multiple orders from the same shop create multiple bills. |
| **BR-12** | When admin converts an order to a bill, ordered stock is deducted from warehouse at that moment. |
| **BR-13** | Direct Shop Sales deduct stock immediately upon bill creation. |
| **BR-14** | Shop advance payments can only be recorded by admin. Order bookers cannot record shop advances. |
| **BR-15** | When any bill is created for a shop with an advance balance, the system automatically deducts the bill amount from the shop's advance balance. Remaining outstanding or remaining advance is clearly shown. |
| **BR-16** | Bills paid in full on the same day by the delivery man are cleared directly by admin. These do not go through the cash recovery tab. |
| **BR-17** | Bills with outstanding amounts are placed in the outstanding pool. Admin assigns individual bills to order bookers for cash recovery independently of route assignment. |
| **BR-18** | ORDER BOOKING vs CASH RECOVERY are two completely independent tasks. Admin can assign routes (for order booking) and recovery bills (for cash recovery) to the same or different order bookers. One order booker can do both tasks in one day: route booking in the morning, then recovery in the afternoon/evening. Or admin can assign them to entirely different people. |
| **BR-19** | Cash recovery assignment is not tied to any route or geography. Admin selects specific bills and assigns them to any order booker regardless of which routes that booker has been assigned. |
| **BR-20** | When admin assigns recovery bills to an order booker: if the booker has already synced for the morning, these assignments will be downloaded on his NEXT sync (can be a mid-day sync or evening sync). If assignment is made before morning sync, it downloads in the morning sync. Admin decides timing — the system supports both scenarios. |
| **BR-21** | Recovery bills must be actioned on the day they are assigned. At midnight, a server-side scheduled job automatically returns all unrecovered bills (where no collection amount was entered) to the outstanding pool. This handles bookers who do not sync in the evening. |
| **BR-22** | Each shop has one ledger. The ledger shows all bills, payments, recoveries, advances, and automatic advance adjustments in chronological order. |
| **BR-23** | Supplier payment flow: advance paid first → stock received later → stock value deducted from advance balance. |
| **BR-24** | Claims from suppliers are recorded with product details and quantities but are NOT added to warehouse stock. Cleared claims add the claim value back to the supplier advance balance. |
| **BR-25** | Salary tracking with partial payment support applies to Salesmen, Order Bookers, and Delivery Men. All managed on the same Staff Salary Management page. |
| **BR-26** | Daily automatic MySQL backup is mandatory. Configurable backup time. Admin can view, download, and restore from the web panel. |
| **BR-27** | Every critical action is logged with date, time, and user for a full audit trail. |
| **BR-28** | All three sales categories are tracked and displayed separately in all dashboards and reports. |

## 6. Business Rules — Day Walkthrough (User Stories)

This section describes a complete working day at Shakeel Traders in sequential order. This is the ground truth for all system behavior.

### 6.1 Morning — Stock Addition from Supplier

#### US-01: Admin receives stock from supplier

Admin opens the web panel and goes to Stock Management → Add Stock from Supplier. Admin selects the supplier company (e.g., CBL). The system shows all products associated with that company. Admin enters received quantities in cartons and loose units and submits. The system deducts the total value from the supplier's advance balance and increases warehouse stock. A stock receipt entry is saved in the supplier's ledger.

**Acceptance Criteria:**
- Admin must select supplier before seeing products
- Quantities entered in cartons and loose units
- Warehouse stock increases immediately on submission
- Supplier advance balance decreases by value of stock received
- Full receipt logged in supplier ledger with date

---

### 6.2 Morning — Order Booker Starts His Day

#### US-02: Order Booker syncs in the morning

The order booker opens the mobile app. He logs in. On the home screen he taps "Sync Now." The app downloads: his assigned routes for today (for order booking), all shops within those routes, all active products with SKU codes and prices, last price charged to each shop, current warehouse stock levels, and any recovery bills that admin assigned before the morning sync.

**Acceptance Criteria:**
- Only routes assigned to this booker for today are downloaded
- Recovery bill assignments available if admin assigned them before sync
- Stock levels reflect current warehouse at time of sync
- All data stored locally for fully offline operation

#### US-03: Order Booker books an order at a shop

The order booker visits a shop on his route. He opens the shop from his route list. He sees all available products searchable by SKU or name. For each product he enters quantity in cartons and loose units. He sees the current price and the last price charged to this shop. If admin has enabled price editing for this shop he can adjust the price within the allowed min/max range. He submits the order. It is saved locally.

**Acceptance Criteria:**
- Products searchable by SKU and name
- Both current price and last-charged price displayed
- Price editing only available if admin enabled it for this shop, within defined percentage range
- Order saved offline, no server connection needed

---

### 6.3 Morning — Salesman Starts His Day

#### US-04: Salesman enters issued stock

The salesman opens the mobile app and logs in. He taps "Sync Now" to get current stock levels. He goes to Stock Issuance. He sees a searchable product list. For each product he plans to take, he enters the quantity in cartons and loose units. He submits the issuance request. Status shows as "Pending Admin Approval."

**Acceptance Criteria:**
- Salesman can only enter quantities, not approve
- Request saved and synced to server
- Status clearly shown as pending
- One issuance request allowed per day

#### US-05: Admin verifies salesman stock issuance

Admin opens the web panel → Stock Management → Pending Issuance Requests. He sees the salesman's submitted request. Admin reviews and clicks Approve. The system immediately deducts the issued quantities from warehouse stock. The issuance is confirmed. Admin tells the salesman he can proceed.

**Acceptance Criteria:**
- Warehouse stock deducted ONLY after admin approval
- Admin can see all pending requests
- Approved issuance is locked and cannot be changed by salesman

---

### 6.4 During the Day — Order Booker (Independent Tasks)

> **IMPORTANT:** Order booking and cash recovery are two fully independent tasks. Admin decides who performs each task and when. The same order booker can do both (route booking in the morning, recovery in the afternoon), or two different order bookers can be assigned each task. The mobile app supports both scenarios.

#### US-06: Admin assigns cash recovery bills to an order booker (flexible timing)

Admin opens the web panel → Cash Recovery → Outstanding Bills. He sees all shops with outstanding bills. He selects specific bills — based on which shops he wants recovered, regardless of geography or routes — and assigns them to a chosen order booker. This assignment has nothing to do with route assignments. The timing of this action is flexible: admin can assign before morning sync (so the booker downloads them in the morning), or after morning sync (so the booker downloads them on a mid-day or evening sync before heading out for recovery).

**Acceptance Criteria:**
- Admin can assign any outstanding bill to any order booker, regardless of route assignment
- Assignment is independent of order booking — different people can be assigned each task
- If assigned before morning sync: booker downloads in the morning
- If assigned after morning sync: booker downloads on next sync (mid-day or evening)
- Assignment is valid for the current calendar day only
- Each bill tracked individually
- At midnight, a server-side scheduled job auto-returns all unrecovered bills to the outstanding pool

---

### 6.5 Evening — Salesman Returns

#### US-07: Salesman enters return stock

In the evening the salesman returns. He opens the app and goes to Stock Return. The system shows his morning issuance list. For each product he enters the quantity he is returning in cartons and loose units. The system automatically displays the calculated sold quantity (issued minus returned). He optionally records cash collected during the day. He submits. Status shows "Pending Admin Approval."

**Acceptance Criteria:**
- Return screen pre-populated with issued products
- Sold quantity = issued minus returned, calculated and shown automatically
- Cash collected is optional input
- Return request synced to server with pending status

#### US-08: Admin verifies salesman return and approves sale

Admin opens web panel → Stock Management → Pending Return Requests. He sees issued quantities, returned quantities, and system-calculated sold quantities with sale value at retail price. Admin can cross-check against physical cash and manually edit the final sale value. Admin clicks Approve. Returned stock is added back to warehouse. Approved sale value is posted to the Centralized Cash Screen.

**Acceptance Criteria:**
- System calculates sale value at retail price per product
- Admin can edit the total sale value before approval
- Returned stock added to warehouse only after admin approval
- Approved sale value appears on Centralized Cash Screen
- All actions logged with date and admin user

---

### 6.6 Evening — Order Booker Returns

#### US-09: Order Booker syncs orders in the evening

The order booker returns and taps "Sync Now." The app uploads all orders booked during the day AND all recovery entries recorded during the day (if any). The server processes each order: products completely out of stock are removed; products with less quantity than ordered are reduced to available quantity. The order booker sees a clear itemized notification of all removed or reduced products. Any recovery bill assigned to him that has no collection entry recorded is flagged — the midnight job will auto-return these to the outstanding pool.

**Acceptance Criteria:**
- Out-of-stock products removed completely
- Partially available products reduced to available quantity
- Booker receives itemized notification of stock adjustments
- All orders marked as "Pending Admin Review" on web panel
- Recovery entries uploaded alongside orders in the same sync
- Midnight server job handles auto-return of unrecovered bills

#### US-10: Admin reviews and converts orders to bills

Admin opens web panel → Order Management. He sees all orders synced from order bookers, filterable by date, booker, and route. For each order he approves, he clicks "Convert to Bill." One bill is generated per order in CBL Salesflo format. At the moment of bill generation, ordered stock is deducted from warehouse. Admin can also view a consolidated Stock Requirement Report per booker.

**Acceptance Criteria:**
- One bill generated per order (not merged per shop)
- Warehouse stock deducted at moment of bill conversion
- Bills are printable in CBL Salesflo format
- Each bill added to the respective shop's ledger
- Bills tagged as "Order Booker" sales type

#### US-11: Admin handles delivery man cash

The delivery man goes out with printed bills. When he returns: for each bill he brings cash for, admin records the outcome. If the shop paid in full: admin marks the bill as cleared and the cash is posted to the Centralized Cash Screen. If the shop paid partially or not at all: the bill moves to the outstanding pool for future recovery assignment.

**Acceptance Criteria:**
- Admin matches cash to specific bills one by one
- Full payment → bill cleared directly, no recovery tab
- Partial or no payment → bill marked outstanding, appears in recovery pool
- Cash received from delivery man recorded on Centralized Cash Screen

#### US-12: Order Booker recovers cash

The order booker has recovery bills assigned to him (downloaded on his morning or mid-day sync). His recovery tab shows all assigned bills with shop name, bill details, and outstanding amount. He visits those shops. At each shop he records the collected amount and payment method (Cash or Bank Transfer). He can record partial or full collection. In the evening he syncs and the recovery data uploads.

**Acceptance Criteria:**
- Recovery tab shows today's assigned bills with full details
- Booker can record partial or full collection per bill
- Data stored offline, synced on next sync
- Payment method recorded (Cash or Bank Transfer)
- Bills with no amount entered: midnight server job auto-returns them to outstanding pool

#### US-13: Admin verifies recovery and updates ledger

Admin opens web panel → Cash Recovery → Pending Verifications. He sees all collections uploaded by order bookers. He verifies each entry. Upon verification, the collected amount is deducted from the outstanding balance of that specific bill in the shop's ledger. Cash is posted to the Centralized Cash Screen.

**Acceptance Criteria:**
- Admin verifies each collected amount individually
- Amount deducted from specific bill's outstanding balance in shop ledger
- Shop ledger updated with recovery entry
- Verified cash appears on Centralized Cash Screen

---

### 6.7 Other Workflows

#### US-14: Admin creates a Direct Shop Sale

Admin opens the web panel → Direct Shop Sales → New Bill. He selects a shop, adds products by SKU or name, enters quantities, confirms. Stock is deducted immediately. A bill is created and added to the shop's ledger, tagged as "Direct Shop Sale."

**Acceptance Criteria:**
- Stock deducted immediately on bill creation
- Bill tagged as Direct Shop Sale
- Bill added to shop ledger

#### US-15: Admin records a Shop Advance

Admin opens web panel → Shop Ledger → [Shop Name] → Add Advance. Enters amount, date, payment method, and optional note. From this point forward, any new bill for this shop will automatically have the advance deducted from its total. Remaining balance is clearly shown on every bill and in the shop ledger.

**Acceptance Criteria:**
- Only admin can record shop advances
- Advance immediately visible as credit in shop ledger
- All future bills for this shop auto-deducted against advance
- Clear balance display on bills and ledger

#### US-16: Admin manages supplier advance and stock receipt

Admin opens web panel → Supplier Management → [CBL]. He records a new advance payment. When stock arrives (see US-01), the system deducts the value from the advance balance automatically. Full history of advances and stock receipts is visible in the supplier ledger.

**Acceptance Criteria:**
- Supplier advances tracked per company separately
- Stock receipt value automatically deducted from advance balance
- Full history with dates visible

#### US-17: Admin records a claim from supplier

Admin opens web panel → Supplier Management → [CBL] → Claims → New Claim. Enters product(s), quantity, date, and reason. Claimed products are NOT added to warehouse stock. When marked as Cleared, the claim value is added back to the supplier advance balance.

**Acceptance Criteria:**
- Claims tracked with product details and quantities
- Claimed products never enter warehouse stock
- Marking claim as Cleared adds value to supplier advance balance
- Full claim history maintained

#### US-18: Admin records salary advance for any staff member

Admin opens web panel → Staff Salary Management → [Salesman / Order Booker / Delivery Man]. Records monthly basic salary. Can add partial advances at any time. Running balance (salary due minus advances paid) is always shown. Month-end clearance is performed by admin. Delivery Men are managed entirely by admin with no system login.

**Acceptance Criteria:**
- Salary tracking supported for Salesman, Order Booker, and Delivery Man
- Partial payments supported with dates
- Running balance always visible
- Month Clearance available at month-end
- Full salary history exportable per staff member

- Today's date and day name

- Number of orders recorded today
- Number of recovery bills assigned today (separate from routes)
- Number of pending sync items (orders not yet synced, recoveries not yet synced)
- Prominent "Sync Now" button
- Last sync date and time displayed

#### 8.2.2 Routes Screen

> **Order Booker:** Displays all routes assigned to this booker today for ORDER BOOKING.

- List of today's assigned routes (for order booking only)
- On selecting a route: list of all shops in that route
- Each shop shows name, address, and visual indicator if it has a recovery bill assigned

#### 8.2.3 Order Booking Screen

> **Order Booker:** Book an order for a selected shop. Accessed by selecting a shop from the Routes Screen.

- Shop name and route shown at top
- Searchable product list (search by SKU or name)
- Per product: SKU, name, current price, last price charged to this shop
- Quantity entry in cartons and loose units
- Price field editable within admin-defined min/max range if price editing is enabled; otherwise read-only
- Order summary shown before submission
- Save order offline — no server connection needed

#### 8.2.4 Recovery Tab

> **Order Booker:** Displays and records cash recovery from shops. FULLY INDEPENDENT from the Routes/Order Booking section.

- List of all bills assigned to this booker for today — regardless of which routes he has been assigned
- Per bill: shop name, bill date, bill number, original bill amount, outstanding amount
- On selecting a bill: enter collected amount, select payment method (Cash or Bank Transfer)
- Partial or full collection supported
- Collected entry saved offline
- Bills with no amount entered will be auto-returned to outstanding pool by midnight server job
- Booker can sync mid-day to download recovery assignments made by admin after morning sync

#### 8.2.5 Sync Screen

> **Order Booker:** Manages morning download and evening upload.

- Morning Sync: downloads routes, shops, products, prices, stock levels, recovery assignments (those made before sync)
- Mid-day Sync: downloads newly assigned recovery bills (admin can assign at any time)
- Evening Sync: uploads all orders and all recovery entries
- Progress shown per record (success / failed)
- After evening sync: displays notification of any products removed or reduced from orders due to stock unavailability
- After evening sync: confirms which recovery entries were uploaded successfully

#### 8.2.6 Today's Summary Screen

> **Order Booker:** Read-only view of everything done today.

- List of all orders booked today (shop, product count, order status)
- List of all recovery entries recorded today (shop, bill, amount collected)
- Sync status of each item

## 9. Non-Functional Requirements

| Requirement | Description |
|---|---|
| **Offline Operation** | The mobile app must work fully offline after morning sync. All data is stored locally (sqflite/Hive) and synced only when connected to the local server. |
| **Performance** | Web panel must load all standard screens in under 3 seconds on the local network. Mobile sync must complete within a reasonable time on local Wi-Fi. |
| **Data Integrity** | Stock can never go negative. All financial entries must have matching ledger records. No orphaned records. Midnight scheduled job enforces bill recovery day-boundary reset. |
| **Security** | All web panel pages require authenticated session. Passwords stored as hashed values (bcrypt or equivalent). All sensitive actions logged with admin user and timestamp. |
| **Backup** | Daily automatic MySQL backup at a configurable time. Admin can restore from backup via web panel. Backup files stored in a dedicated local folder. |
| **Scalability** | System must support at least 10 order bookers, 10 salesmen, 10 delivery men, 500 shops, 200 products, and 1000 orders per day without performance degradation. |
| **Print Compatibility** | Bills must print in CBL Salesflo format. Company profile info (name, logo, address, NTN) must appear on all printed bills. Reports must be exportable to Excel. |
| **Audit Trail** | Every create, update, approve, and verify action is logged. Audit log accessible to admin and filterable by user, date, and entity type. |
| **Midnight Scheduler** | A server-side cron job runs at midnight daily. It: (1) auto-returns all unrecovered bill recovery assignments to the outstanding pool, (2) flags any order booker who had assignments but did not sync. Admin sees these flagged items in the dashboard. |
